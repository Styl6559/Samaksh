# Samaksh Algorithms & Logic

The Samaksh prototype utilizes several specialized algorithms and logic flows to ensure robustness, accessibility, and real-time performance.

## 1. Ambient Noise Rejection Algorithm (Frontend)

To prevent the system from taking accidental photos due to background noise (e.g., a TV or a conversation), the frontend employs a strict keyword-filtering algorithm over the continuous Web Speech API stream.

**Logic Flow:**
1. Capture raw speech transcript.
2. Strip punctuation (`.replace(/[.,!?]/g, '')`) generated by the mobile OS.
3. Classify transcript into specific buckets (`read`, `location`, `general`).
4. **Result:** If the transcript does NOT contain specific hardcoded trigger words (e.g., "where", "read", "describe"), the command is instantly dropped and ignored. This prevents the Pi from wasting API limits on ambient noise.

## 2. Audio-Feedback Infinite Loop Prevention (Frontend)

A critical accessibility bug in Voice-UI systems is the "infinite loop," where the microphone picks up the system's own Text-To-Speech (TTS) output and interprets it as a new command.

**Logic Flow:**
1. Check `window.speechSynthesis.speaking` boolean.
2. If the AI is actively speaking, all incoming microphone transcripts are dropped.
3. **Exception:** The algorithm includes a bypass for interruption commands (`"stop"`, `"shut up"`), allowing the user to halt the TTS engine even while the loop prevention is active.
4. **Result:** The system never "hears itself," ensuring clean input pipelines.

## 3. Atomic State Machine (Database / Edge Sync)

Because the frontend and backend operate asynchronously over the internet, race conditions can occur (e.g., the user spamming a command before the camera initializes). We implemented an Atomic State Machine in Firebase.

**Logic Flow:**
1. Website pushes a command object to Firebase: `{ type: "read", status: "pending" }`
2. Raspberry Pi WebSocket listener triggers on the `pending` state.
3. Pi instantly updates the database to `status: "processing"`.
4. While `status === "processing"`, the React frontend strictly blocks any new camera commands from being sent.
5. Pi captures image, calls Gemini, pushes the text response, and updates the state to `status: "done"`.
6. **Result:** Exactly ONE picture is taken per command, completely eliminating overlapping hardware requests and duplicate API calls.

## 4. Asynchronous Navigation Threading (Backend)

The Navigation Mode requires the Raspberry Pi to capture an image autonomously every 10 seconds, but the system must still be able to listen for "stop" or mode-switch commands via Firebase WebSockets.

**Logic Flow:**
1. When Mode changes to `navigation`, the Pi spawns a separate Daemon Thread (`threading.Thread`).
2. The main thread continues running the Firebase WebSocket listener, remaining highly responsive.
3. The background thread uses a hardware frame-clearing loop to dump stale USB webcam buffers (`camera.read()` x3) before grabbing a fresh frame.
4. The thread calls Gemini and pushes the response to the user.
5. A `threading.Event()` flag is used so that if the user says "mode normal", the main thread sets the flag, and the background thread instantly halts its 10-second sleep cycle and safely dies.
6. **Result:** Non-blocking, continuous environmental scanning without freezing the hardware controller.
